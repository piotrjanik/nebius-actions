# Bucket-mounted Axolotl config — design

Date: 2026-06-30
Status: approved (design); pending implementation plan

## Problem

The `demo-run-job` workflow currently ships the Axolotl `config.yaml` to the job by
base64-embedding `examples/axolotl/config.yaml` into the container `--args`. This avoids
needing an Object Storage bucket, but it does not demonstrate the bucket-mount path that
real fine-tunes use to read configs and persist artifacts.

We want to evolve the demo so it:

1. **Inlines** the Axolotl config directly in the workflow (no separate vendored file).
2. **Uploads** it to a pre-existing Nebius Object Storage bucket via a **new reusable
   action**.
3. **Mounts** that bucket into the job, which then reads the config from the mount.

## Key constraints (verified against the live `nebius` CLI)

- `nebius storage bucket` is **control-plane only** (create/delete/list the bucket
  resource). There is **no** data-plane `put-object`/`cp`. Object bytes must go over the
  **S3 API** (SigV4), which needs an AWS-style **access-key + secret** — the bearer token
  from the SA key does not work for the S3 data plane.
- The job mount flag is:
  `--volume s3://BUCKET:/container_path[:MODE[:PROFILE]]`, `MODE` ∈ `rw|ro` (default `rw`).
  `PROFILE` is an AWS credentials profile (default `default`); for Nebius secret-manager
  auth the form is `PROFILE@SECRET_NAME_OR_ID_OR_VERSION_ID` ("MysteryBox" auth).
- `nebius iam v2 access-key create` supports `--account-service-account-id`,
  `--parent-id`, `--expires-at`, and `--secret-delivery-mode {inline|mystery_box|explicit}`.
  This lets us mint a **short-lived** access key **from the SA we already have** and have
  Nebius deliver its secret **into MysteryBox** in one call.

## Decisions

- **S3 credential source:** mint an **ephemeral** access key from the configured service
  account at upload time. The SA key (already set up by the `setup` action) stays the only
  long-lived root secret; the minted key self-expires via `--expires-at`, so no explicit
  revoke is required.
- **Job mount auth:** MysteryBox secret, referenced as `…:ro:default@<secret-id>`. Nothing
  S3-specific is baked into the job image or printed into the job spec.
- **Bucket lifecycle:** the bucket **pre-exists** and is passed in (e.g.
  `vars.NEBIUS_CONFIG_BUCKET`). The new action uploads only; it does not create buckets.
- **S3 client:** add `@aws-sdk/client-s3` and point it at the Nebius S3 endpoint. It bundles
  into `dist/` via ncc (hermetic — no reliance on a preinstalled `aws` CLI), consistent with
  how this repo vendors its dependencies.

## End-to-end flow

```
workflow step (heredoc)        new action: upload-object               submit-job (mounts)
───────────────────────        ──────────────────────────────         ──────────────────────
inline config.yaml ──file──► 1. mint ephemeral access key
(written to runner)             nebius iam v2 access-key create
                                  --account-service-account-id <SA>
                                  --parent-id <project>
                                  --expires-at <now+TTL>
                                  --secret-delivery-mode mystery_box
                             2. fetch plaintext creds ──► S3 PutObject
                                                          s3://<bucket>/<key>
                             3. outputs: object-uri, secret-id ────────►  --volume
                                                              s3://<bucket>:/workspace/cfg:ro:default@<secret-id>
                                                              --args  axolotl train /workspace/cfg/config.yaml
```

## New action: `upload-object`

A generic, reusable "upload a local file to a Nebius Object Storage bucket" action, built to
match existing repo conventions: a TypeScript entrypoint bundled to a committed `dist/`.

Files:
- `src/entrypoints/upload-object.ts` — entrypoint (reads inputs, orchestrates, sets outputs).
- `src/core/storage/` — new module:
  - `keys.ts` — pure builder for the `iam v2 access-key create` args (`mintEphemeralKeyArgs`).
  - `s3.ts` — thin `putObject()` wrapper over `@aws-sdk/client-s3` (endpoint, region, creds).
  - `index.ts` — barrel.
- `actions/upload-object/action.yml` + `actions/upload-object/dist/` (generated by `scripts/build.mjs`).
- `__tests__/storage/storage.test.ts` — unit tests for the pure arg-builder and key/URI formatting.

### Interface (`action.yml`)

```yaml
inputs:
  source:              # local file path on the runner (required)
  bucket:              # target bucket name (required)
  key:                 # object key / prefix within the bucket (required)
  service-account-id:  # SA to mint the ephemeral key for (required)
  project-id:          # parent for the access-key + MysteryBox secret (required)
  expires-in:          # ephemeral key TTL (default: 2h)
  endpoint:            # S3 endpoint (default: Nebius eu-north1 storage endpoint)
  region:              # S3 region (default: eu-north1)
outputs:
  object-uri:          # s3://<bucket>/<key>
  secret-id:           # MysteryBox secret id, for the job's :default@<id> mount
```

### Internals (3 small, independently testable units)

1. `mintEphemeralKey(spec)` — builds + runs `nebius iam v2 access-key create … --format json`,
   parses the JSON, returns `{ awsAccessKeyId, awsSecretAccessKey, secretId }`.
   - **Plaintext-for-upload path (to pin in the plan):** the runner needs the plaintext
     secret to perform the S3 `PutObject`. Preferred: `--secret-delivery-mode mystery_box`
     and, if the create response does not echo the plaintext, fetch it via
     `nebius mysterybox payload` / `iam v2 access-key get-secret`. Fallback if neither
     returns plaintext for a MysteryBox-delivered key: use `--secret-delivery-mode inline`
     (plaintext returned to the runner) and **separately** create a MysteryBox secret holding
     the creds for the job mount. Either way the public contract (`secret-id` output +
     ephemeral, self-expiring key) is unchanged.
2. `putObject({ endpoint, region, creds, bucket, key, body })` — single `PutObjectCommand`
   over `@aws-sdk/client-s3`.
3. Entrypoint composes them and emits `object-uri` + `secret-id`.

The arg-builder is pure (no network), mirroring `buildCreateJobArgs` in
`src/core/jobs/jobs.ts`, so it is unit-testable without live calls.

## Workflow changes (`.github/workflows/demo-run-job.yml`)

- Replace the `Encode Axolotl config` step with a heredoc step that writes the Axolotl YAML
  **inline** to `$RUNNER_TEMP/config.yaml`.
- Delete `examples/axolotl/config.yaml` (now inlined) and rewrite the workflow header comment
  block, which today sells the "no bucket needed" trade-off.
- Add an `upload-object` step after `setup`:
  - `source: ${{ runner.temp }}/config.yaml`, `bucket: ${{ vars.NEBIUS_CONFIG_BUCKET }}`,
    `key: <run-scoped>/config.yaml`, `service-account-id`, `project-id`.
- `submit-job`: add
  `mounts: ${{ vars.NEBIUS_CONFIG_BUCKET }}:/workspace/cfg:ro:default@${{ steps.upload.outputs.secret-id }}`
  and change `--args` from the base64-decode pipeline to `axolotl train /workspace/cfg/config.yaml`.
- The Axolotl config itself is unchanged: it still writes adapters to `/tmp/output`.
  Persisting **output** to a bucket is explicitly **out of scope** for this change.

## Testing & docs

- Vitest unit tests for the `storage` arg-builders and key/URI formatting, following
  `__tests__/jobs/jobs.test.ts`. No live network.
- `npm run all` (lint + typecheck + test + build) must pass; the committed `dist/` for the new
  action is required (no `.gitignore` of `dist/`).
- Update `README.md` actions table with an `upload-object` row and a short usage note.

## Out of scope (YAGNI)

- Creating/managing the bucket (assumed pre-existing).
- Persisting Axolotl **output** to a bucket.
- Multi-file / recursive uploads (single file is enough for the config use case).

## New configuration required

- `vars.NEBIUS_CONFIG_BUCKET` — the pre-existing Object Storage bucket name.
- The configured service account needs permission to create IAM access keys and MysteryBox
  secrets, and read/write the bucket. No new GitHub **secrets** are introduced.
